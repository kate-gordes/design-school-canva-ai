# Accessibility Testing Checklist

Canva’s mission is to make design accessible to anyone, regardless of ability, device, or internet. In line with this mission, Canva strives to meet _Web Content Accessibility Guidelines (WCAG) 2.2_ at the AA level and go beyond. This checklist outlines key areas and tests to ensure new features or designs are accessible. Not all tests apply to every project, so include the relevant ones for your “test party” (i.e. the scope of features under test). By incorporating these checks, you help ensure an ultra-inclusive experience for all users.

## 1. Color Contrast

All text and important visual elements should have sufficient contrast against their background. **What to do:** Check that the contrast ratio between foreground text (or UI component) and its background meets accessibility standards. For normal-sized text, WCAG Level AA requires a minimum contrast ratio of 4.5:1 (and 3:1 for large text). This helps users with low vision or color blindness read content and identify interface elements that might otherwise blend into the background.

- _How to test:_ Use a color contrast analysis tool such as the **Color Contrast Analyzer (CCA)** or the **WebAIM Contrast Checker**. These tools will calculate the contrast ratio and indicate if it passes AA guidelines. Ensure that **text**, icons, and interactive graphics all meet the required contrast thresholds. This includes borders or focus outlines on form fields and buttons, which should also be clearly visible against adjacent colors.
- _Expected result:_ Every piece of text or essential UI icon should have contrast at least at the 4.5:1 ratio (for normal text) or 3:1 (for large text or UI graphical components) to be compliant. Users with visual impairments should be able to distinguish text and interface elements without strain. If any text or icon is purely decorative and conveys no information, it can be excluded from contrast requirements, but generally **higher contrast is better** for readability.

## 2. Keyboard Navigation

All functionality must be operable via keyboard alone, without requiring a mouse or touch. **What to do:** Unplug your mouse (or simply ignore it) and navigate through the interface using the keyboard only. Try to accomplish all the main user flows (e.g. navigating menus, activating buttons/links, filling forms) using Tab, Shift+Tab, arrow keys, Enter/Space to activate controls, and Escape where applicable. According to WCAG, every interactive element and control should be reachable and operable with a keyboard interface. This ensures that people who cannot use a mouse (e.g. blind users or those with motor disabilities) can still use the product.

- As you navigate, verify that a **visible focus indicator** is present on the element you have tabbed to (e.g. a highlight or outline on links, buttons, inputs). It should be obvious which component currently has keyboard focus. Do not remove or hide the default browser focus outline without providing an equivalent visible style.
- Ensure the **focus order** is logical and follows the visual reading order. Tabbing should move through the UI in a sensible sequence (e.g. left-to-right, top-to-bottom) and not jump around unexpectedly.
- Check that there are **no keyboard traps** – meaning you should never get stuck on an element with no way to move focus away. Every interactive component should allow focus to move forward (with Tab) or backward (with Shift+Tab) or have an obvious way to exit (for components like modals or dropdowns, pressing Escape should close them).
- All interactive features (buttons, sliders, dialogs, etc.) should be usable via keyboard. If a component involves complex pointer interactions (drag-and-drop, swiping, etc.), there should be an alternative keyboard method. _For example:_ if an item can be dragged with a mouse, the interface could also allow using arrow keys to move it or provide “Move Up/Down” buttons, so that **using a mouse is not the only way to complete the task**.

_Expected result:_ A keyboard-only user can fully operate the feature or page. **Focus never disappears** and is always visible as you tab through. The tab order is intuitive, following a meaningful sequence through content. All interactive controls respond to keyboard events (Enter/Space to activate, arrow keys for menus or sliders, etc.), and nothing requires an exclusively mouse-driven action. By meeting these conditions, the app adheres to the WCAG requirement that all functionality be accessible via keyboard, ensuring inclusivity for users who rely on assistive technology or cannot use a mouse.

## 3. Motion, Animations and Audio

Moving or multimedia content should not interfere with usability or trigger health issues. **What to do:** Identify any elements in your test scope that involve **motion**, **animations**, **video**, **GIFs**, or **auto-playing audio**. For each, ensure they meet accessibility criteria:

- **No distracting animations:** If an animation or video plays automatically, check its duration. Auto-playing content that lasts more than 5 seconds must provide a mechanism for the user to pause, stop, or hide it. This prevents distractions and gives users control. If the animation is purely decorative or short (5 seconds or less), it’s generally okay as is, but anything longer needs user controls.
- **Avoid flashing content:** Content that flashes or blinks can cause seizures in sensitive individuals. Verify that nothing on screen flashes more than 3 times per second, **unless** the flashing is so low-impact (small size and low contrast) that it falls below known seizure risk thresholds. It’s safest to avoid any high-frequency flashes altogether.
- **Audio control:** If audio plays automatically (e.g. background music or a sound effect on load), the user must have a way to stop or mute it if it continues beyond a few seconds. Users who rely on screen readers can be severely impacted by unexpected audio, so auto-audio is typically discouraged unless brief or with controls.
- **Reduced motion:** Some users enable “reduced motion” or similar settings to avoid parallax and animations (because they may get motion sickness or distraction). Test with any _reduced motion_ setting (in the OS or browser) to ensure animations are minimized or disabled accordingly. For example, in macOS or iOS enabling “Reduce Motion” should prevent any non-essential animations in Canva’s interface (if applicable).

_Expected result:_ Motion and multimedia content should be designed to be inclusive. Users should not be subjected to unmanageable flashing or auto-playing media. Any motion effect that could be problematic should either stop quickly or offer an accessible control to pause it. By providing play/pause controls for media and avoiding intense flashes, you conform to WCAG guidelines such as **Pause, Stop, Hide** for content that moves or auto-updates and **Three Flashes or Below** for flashing content. This ensures users prone to photosensitive seizures or easily distracted by motion can safely use the product.

## 4. Adaptability to User Settings (Zoom, Fonts, etc.)

Your design should **adapt to various user preferences and system settings** without breaking functionality. Many users override default display settings to make content easier to consume (e.g. zooming in for larger text, using high contrast mode, or increasing default font size). **What to do:** Simulate common adjustments and verify the interface remains usable:

- **Zoom to 400%:** In a desktop browser, zoom the page up to 400%. At 400% zoom, users with low vision effectively see the page as if through a very narrow viewport (roughly equivalent to a mobile screen width). Check that content reflows properly: you should be able to scroll vertically to read content, but horizontal scrolling should not be necessary for reading a line of text. All important information and actions should remain available. _Expected:_ No loss of content or functionality occurs at high zoom; the layout may simplify into a single column but users can still perform all tasks.
- **Increase text size:** Some users set a larger base font in their browser or OS. Test by enlarging just the text (e.g. in browser settings, increase font size to 200% if possible) rather than full zoom. Ensure that text enlargement alone doesn’t cut off or overlap content. Interfaces should be able to handle at least **200% text size increase** without truncation, or provide scrolling if absolutely necessary (per WCAG 1.4.4 Resize Text).
- **High Contrast or custom color schemes:** If your OS has a High Contrast mode or custom color scheme, try enabling it. The app should still display content correctly (perhaps with default system colors overriding some styles). Verify that important information isn’t lost (for example, if your design relied solely on color coding, ensure there are alternative indicators, since high contrast mode might override colors).
- **Reduced Motion setting:** As noted earlier, enabling a system “reduce motion” setting should minimize non-essential animations. Verify that any animated effects (background transitions, parallax, etc.) are subdued when this setting is on. Also, if there are videos set to autoplay, check that they do **not** autoplay when reduce-motion is on (if possible), or at least they honor the earlier rule of providing a pause control.

_Expected result:_ The application should be robust against different user settings. Content should **reflow** gracefully at extreme zoom levels (up to 400%) without requiring two-dimensional scrolling. Large font sizes should not break the layout or hide content. In high contrast modes, all text is still visible (no white text on white background issues, for example) and controls remain distinguishable. By testing these scenarios, we ensure compliance with adaptability criteria like _Reflow_, _Resize Text_, and _User Preferences_, which help users with low vision, cognitive considerations, or other needs to use the product in their preferred way.

## 5. Headings and Structure

Use a clear heading structure to organize content, and ensure labels for controls are provided where needed. **What to do:** Review the page or feature for semantic structure:

- All pages or views should have a **descriptive H1 heading** that gives the primary title or purpose of the page. Subsequent sections should be introduced with hierarchical headings (H2 for major sections, H3 for subsections, etc.). Make sure you aren’t just styling text to look big or bold, but actually using heading tags in code. For example, text that visually looks like a section title should be marked up as a heading, not just a larger font paragraph.
- Check that **no important content is unheaded.** Every major content region should fall under some heading. It’s a common best practice that users (especially screen reader users) can navigate via headings to understand the page’s outline. If you find a section of content that doesn’t have a heading when it logically should, that’s an issue to fix (either add a heading or reconsider the structure).
- **Order of headings:** Verify the heading levels are used in logical order (e.g., H1 followed by H2s, then H3s within a section, etc., without jumping arbitrarily). Skipping heading levels (like going from an H1 straight to an H4 for styling reasons) can confuse assistive tech. Headings should form a nested outline of the content.
- **Form labels and names:** Although not exactly about headings, it’s related to structuring content. Ensure that any form fields (inputs, dropdowns, etc.) have an associated label that is announced to screen readers. The label can be a visible `<label>` or an ARIA-label if a visual label isn’t present. Similarly, any interactive icon (like an icon-only button) should have an accessible name (via an `aria-label` or equivalent) describing its function. This falls under WCAG’s “Headings and Labels” criterion – meaning not only headings for sections but also clear labels for UI controls. For example, if there’s a button with just a magnifying glass icon, ensure it has alt text or an aria-label like “Search”.

_Why it matters:_ A well-structured page with headings allows users of assistive technologies to navigate efficiently. Many screen reader users pull up a list of headings to jump to the content of interest. If headings are missing or improperly nested, this experience is impaired. Likewise, labels ensure that form controls or icons are not ambiguous – a screen reader should announce what a button or field is for, based on a proper label. By following proper heading hierarchy and labeling, you adhere to WCAG guidelines that emphasize info and relationships (all content must be programmatically structured) and the specific success criteria for headings and labels (Level AA) which require descriptive headings and form labels. Overall, this makes content more **navigable and understandable** for everyone.

## 6. Alternative Text for Images and Icons

All non-text content (images, graphics, icons, media) should have a text alternative that conveys its meaning. **What to do:** Go through every image or graphic in your test scope:

- If an image is **decorative** (purely for layout or visual flair with no informational content), it should be marked as such (e.g. have an empty `alt=""` attribute or be hidden from assistive tech) so that it doesn’t distract users with a screen reader.
- If an image or icon **conveys information or has a function**, it **must have appropriate alternative text**. This could be a concise alt attribute on an `<img>` tag, or if the image is used as a button or link, the button/link should have an accessible name that describes its purpose. For example, an “Edit” pencil icon button should have `aria-label="Edit document"` or similar, so that a screen reader user knows what the icon does.
- **Meaningful alt text:** The alternative text should capture the _essence_ of the image’s content or function without being overly verbose. It’s not usually necessary to say “Image of...” in the alt text (screen readers announce it’s an image). Instead, describe what the image is or what information it’s conveying. If the image is a graph or complex graphic, provide a summary of the key insight. If it’s a photo of a person or object and is relevant, mention who or what it is. _Example:_ If there’s an image showing a new interface feature, alt text might be “Screenshot of the new color picker interface,” which gives the user context.

_Expected result:_ Every meaningful visual element has a text equivalent. You can test this by using a screen reader or an accessibility inspector: verify that all images are either ignored (if decorative) or have descriptive alt text announced. This ensures users who cannot see the image aren’t missing out on information. As WebAIM notes, alternative text is read by screen readers in place of images, allowing users with visual impairments to perceive the content and purpose of the image. Moreover, having alt text means if images fail to load, the browser shows the alt text so users still know what was intended. **No image should be simply silent or unlabeled** in the accessibility tree. By providing good alt text, we comply with WCAG’s requirements for text alternatives (1.1.1) and make sure no one is left wondering what an important graphic is conveying.

## Additional Resources and Help

Accessibility is a broad topic, and it’s normal to have questions. Here are some additional resources and support channels available:

- **General Accessibility Guidance:** Refer to Canva’s internal [Accessibility Fundamentals documentation](https://docs.canva.tech/easel/foundations/fundamentals/accessibility/) for a deeper understanding of Canva’s accessibility standards and guidelines. This document covers the principles and best practices that Canva follows to make products accessible.
- **Slack Channel – #help-accessibility:** For any questions, clarifications, or to seek advice on accessibility issues, you can reach out in the Slack channel [#help-accessibility](https://canva.enterprise.slack.com/archives/C074AJLP8T1). The accessibility team and knowledgeable peers monitor this channel, making it a great place to get quick help or discuss solutions.
- **Accessibility Office Hours:** Canva’s Accessibility team holds weekly office hours sessions. You can book a slot to discuss any accessibility challenges or review features for accessibility. Details about the schedule and how to join can be found on the [Accessibility Office Hours Confluence page](https://canvadev.atlassian.net/wiki/spaces/A11Y/pages/3890020549/Accessibility+Office+Hours). This is an excellent opportunity to get live feedback or hands-on guidance for your specific project.

By utilizing these resources, you can continuously improve your understanding and implementation of accessibility. **Remember, accessibility is an ongoing team effort** – don’t hesitate to seek help and share knowledge. Every improvement you make using this checklist and the above resources brings Canva closer to its mission of empowering everyone to design and collaborate without barriers.
